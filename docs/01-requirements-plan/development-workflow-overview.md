# 408Master 开发流程总览

## 项目目标

408Master 的目标不是简单换皮原考试系统，而是在 408 真题刷题基础上，加入可解释、可追踪、可反馈的智能学习能力：

- 建立 2011-2024 年 408 真题题库。
- 建立 408 知识点和方法论知识库。
- 使用 RAG 为 AI 解析提供依据，减少幻觉。
- 提供常规、费曼学习法、第一性原理、柏拉图式对话四种解析视角。
- 记录学生做题、反馈和学习过程，形成学生画像。
- 支持 Web 学生端、管理端、小程序端和云端部署。

## 推荐开发顺序

1. **稳定数据底座**：先确保数据库结构、真题数据、图片资源和知识点数据可靠。
2. **跑通基础业务**：学生端试卷中心、答题、批改、记录、管理端接口不报错。
3. **接入 AI/RAG**：先做检索和引用，再做四种 Skill 风格，最后做反馈闭环。
4. **补齐 Agent 记录**：把学习事件、学生画像、反馈数据写入数据库。
5. **端到端验证**：Web、小程序、本地接口、构建、部署文档逐项验证。
6. **整理文档和汇报**：把开发记录、问题排查、部署经验和协作过程归档。

## 具体开发流程

### 1. 开分支

较大改动使用独立分支，例如：

- `feature/*`：新功能。
- `refactor/*`：框架迁移或结构重构。
- `deploy/*`：部署相关。
- `codex/*`：Codex 辅助完成的阶段性工作。

### 2. 修改与验证

每次改动尽量带上对应验证：

- 后端：`mvn package -DskipTests`
- 前端：`npm run build`
- Python 脚本：`python3 -m py_compile ...`
- 数据库：导入顺序和关键计数 SQL
- 接口：登录、试卷、AI styles、RAG debug、知识图谱等

### 3. 记录问题

遇到问题不要只修代码，最好同步记录：

- 问题现象。
- 触发路径。
- 根因。
- 修复方式。
- 如何验证。

适合放入 `../03-engineering-experience/` 或 `../02-work-records/`。

### 4. 合并和回顾

使用 PR 合并，PR 描述至少包含：

- 本次目标。
- 主要改动。
- 验证方式。
- 已知风险。
- 后续任务。

合并后把重要节点写入 `../02-work-records/`，形成可以复用的项目经历。
